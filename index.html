Usage Guide

How to read and interact with each panel of the dashboard.

Search Tree

Each node represents one deduce' invocation (one goal). Parent→child edges show the subgoals produced by a rule application.

Node label format Rule (i/m #k) [children] duration
Rule The inference rule applied (e.g. ΣI, Var, EFQ)
i/m AND position — the i-th subgoal out of m produced by the parent rule (all must succeed)
#k OR index — which alternative branch (any one suffices). Omitted when only one branch exists
[children] Total number of direct children, including collapsed ones
duration Wall-clock time for this subtree (in ms)

Colors: success   failure (all rules exhausted)   depth-bound exhaustion   loop avoidance   time limit

Interaction: Click a node to expand or collapse its subtree. Nodes at depth 4+ are collapsed by default.

Flame Graph

Each cell corresponds to one goal node. The width is proportional to the wall-clock time spent in that subtree. The vertical axis represents recursion depth (root at the bottom).

Wide cells are temporal hotspots — look for them to find where the prover spends most of its effort. Colors follow the same scheme as the Search Tree.

Rule Statistics

A table grouping every search-tree node by the rule that produced it.

Att.Total number of attempts for this rule
Suc.Successful discharges
FailFailures (all sub-rules exhausted)
Dep.Terminated by depth bound
LoopTerminated by loop avoidance
TimeTerminated by time limit
Pend.Still pending (incomplete search)
RateSuccess rate (Suc. / Att.)
Tot. msTotal wall-clock time spent on this rule

Interaction: Click any column header to sort the table by that column.

Failure Analysis

Failure events are grouped by category (Deduce Failed, Depth Exceeded, Loop Avoided, Time Limit). Within each category, identical goals are merged with a count.

Number badge How many times this exact goal failed in this category
d=N The recursion depth at which the failure occurred
Goal text The failed goal in DTT form (context ⊢ target type)

High counts on a single goal indicate that the prover repeatedly hits the same dead end — a strong signal for where a learned ranker could prune.

Run It Yourself

A pre-built Docker image bundles the compiled lightblue binary, the KWJA tokeniser models, and the pre-baked JSeM 720 proof-search query. No external services or datasets are required at runtime. 

# No build needed — pull and run in one command
docker run -p 3000:3000 lilkohapizza/wani-viz-demo

# → open http://localhost:3000/searchlog

What runs at startup (default): the container loads the pre-baked JSeM 720 ProofSearchQuery into Express, starts the Yesod server on port 3000, and a visitor of /searchlog sees the four-panel dashboard rendered from a real proof-search trace.
